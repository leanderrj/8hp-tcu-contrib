# Can_ZF8HP — Stm32-vcu integration patch

DBC-driven codec for the ZF 8HP TCU ↔ VCU CAN protocol. The C99 encoder/decoder is generated by `cantools` from `proposals/dbc/zf8hp-tcu.dbc`; a thin C++ wrapper bridges into the VCU.

## Files

```
include/zf8hp_tcu.h              (new — generated, MIT-licensed)
include/Can_ZF8HP.h              (new — C++ wrapper)
src/zf8hp_tcu.c                  (new — generated, MIT-licensed)
src/Can_ZF8HP.cpp                (new — C++ wrapper)
test/test_Can_ZF8HP.cpp          (new)
```

The cantools output retains its original MIT license header. MIT is GPL-compatible (more permissive), so redistributing it inside GPLv3 Stm32-vcu is fine; the headers must stay intact.

## Regenerating `zf8hp_tcu.{h,c}`

The C99 codec is **generated**, not hand-written. Schema changes go in the DBC, not in the C source. Regenerate with:

```bash
python3 -m venv venv && source venv/bin/activate
pip install cantools
bash proposals/firmware/can_codegen/generate.sh
```

Hand edits to `zf8hp_tcu.{h,c}` will be overwritten. All glue logic — counters, clamping, wrapper API, latching — lives in `Can_ZF8HP.{h,cpp}`, which is hand-written and stable.

## Why this design

- **Single source of truth.** The DBC is the protocol. Both sides of the link (VCU consumer and the not-yet-written TCU firmware) consume the same generated C from the same DBC. Schema drift becomes a compile error rather than a documentation bug.
- **Testable on the host.** The codec is plain C99 with `<stdint.h>` and `memset` — no malloc, no RTOS, no peripherals. Drop it on a Mac/Linux box and `make Test` runs the full encode/decode + clamp + counter coverage in milliseconds.
- **Rebuildable in CI.** `generate.sh` is one command; the resulting C is deterministic given a fixed `cantools` version (we used 41.3.1). Future contributors regenerate, build, run tests — same flow as ours.

## What's tested

- Round-trip encode → decode of every signal in `VCU_GearRequest`.
- Status1 decode populates the latched VCU view of gear / RPMs / oil temp / line pressure with correct scaling.
- Pump status decode latches and flips `PumpFrameSeen()`.
- Unknown CAN IDs are silently ignored (not counted as decode errors).
- Short DLCs are rejected with `decodeErrors++`.
- `SetAccelPedal()` clamps `< 0` to 0 and `> 100` to 100, scales by 0.5.
- The TX counter advances 0..15 modulo 16 between successive packs.

24 assertions, run as part of `make Test` on the host.

## What's *not* tested here (deliberately)

- LIN aux pump producer side (separate work — that's the TCU firmware, not the codec).
- Real CAN hardware. A `CanHardwareLinux` backend (target #4 in the wider plan) would let us replay real captures end-to-end.
- The factory BMW PT-CAN integration. Out of scope for the OI↔TCU protocol.
